About me

Hi there, my name is Rodrigo, and I am a Software Developer and former Electrical Engineer.

I have a passion for technology and problem-solving, so I chose to pursue a career in software development.

I have experience in both front-end and back-end development, utilizing a range of technologies and frameworks. On the front-end side, I am proficient in Bootstrap, HTML, CSS, JavaScript, and JQuery but most of my recent projects are now built using TypeScript, React.js, and React-Native for mobile applications. I also contributed with a few projects using Next.js, Apollo client and GraphQL

For back-end development, I have some experience with Java, PHP, C# and Node.js, with additional knowledge of the frameworks Nest.js, Express.js, and Laravel.

Furthermore, I am well-versed in working with various types of databases, including relational databases like MySQL, PostgreSQL, as well as non-relational databases like MongoDB, and Google Firestore. I am also familiar with Object-Relational Mapping (ORM) with database agnostic implementations.

In my free time, I enjoy staying updated with the latest technologies and trends in the software development industry. I also like to read self-help books and constantly work on improving my communication skills.

Good Meal (Meal Planner)

I started helping developing this project as a volunteer and I was hired after a couple months to speed up the project delivery. Mealplanner is essentially a 3-tier application with a UI (or front end) that executes entirely in the client's browser, a middleware server that presents a single graphql API and postgres database backend. The front end is a React app based on React Hooks and React Material UI components. The middleware is Postgraphile which is a NodeJS service that automatically generates a graphql API and schema from the structure of a relational database. Finally the application's database schema is hosted in postgres. More on each of these component tiers below.

Robo Friends

This application utilizes the JsonPlaceholder API to retrieve information for 10 fictitious users, storing the data in a state variable. Subsequently, I designed a card component to present each user's details and incorporated another API, RoboHash. By passing the user ID to RoboHash, the API generates a unique robot image for each user. To consolidate the user display, I implemented a cardList component within the application's main body. The integrated search bar facilitates user filtering, rendering only those individuals whose information matches the input.
